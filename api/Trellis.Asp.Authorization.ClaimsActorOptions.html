Remarks

Default behavior maps the OIDC standard sub claim to Id and the permissions claim to Permissions. Override the claim names to match your identity provider's token format.

Nested-claim mapping (ga-15). Both ActorIdClaim and PermissionsClaim are matched against the flat Type string only — no JSON-path or dotted traversal is performed. Two practical consequences:

  • Provider-prefixed claim names work as-is. Set the option to the full claim type the JWT handler emits, e.g. "http://schemas.microsoft.com/identity/claims/objectidentifier" (Entra oid) or "extension_role" (Entra External Identities custom attribute). Whatever the issuer/handler exposes via ClaimsIdentity.FindFirst(name) is what to put here.
  • JSON-object claims are not auto-flattened. When a JWT contains a nested object (e.g. { "app_metadata": { "roles": [ ... ] } }), the JWT handler stores the value as the raw JSON string under claim type "app_metadata". To project nested values into Permissions, subclass ClaimsActorProvider and override GetCurrentActorAsync(CancellationToken) to parse the JSON yourself, or use EntraActorProvider with a custom MapPermissions delegate which receives the full Claim sequence.
  • Multi-valued JWT claims are flattened automatically by JwtBearerHandler — a JSON array such as "permissions": [ "orders:read", "orders:write" ] arrives as multiple Claim instances of the same Type, which the default provider already aggregates via FindAll.
